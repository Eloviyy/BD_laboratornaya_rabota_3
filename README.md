# BD_laboratornaya_rabota_3
Выполнение лабораторной работы №3 по дисциплине Технология Big Data.
«АВТОМАТИЗИРОВАННАЯ РАБОТА С ХРАНИЛИЩАМИ ДАННЫХ С ИСПОЛЬЗОВАНИЕМ ФРЕЙМВОРКА BUILD DATA TOOLS»
Цель: научиться автоматизировать работу с хранилищами данных с использованием фреймворка build data tools.
Задачи:
1.	развернуть приложение dbt;
2.	разработка скриптов автоматизированной работы с данными;
3.	поставить работу скриптов на расписание.
Ход лабораторной работы
Задача 1. Развернуть приложение dbt.
Для выполнения работы был выбран dbt Core версии 1.8.2, функционирующий под управлением Python 3.12.3 в изолированном виртуальном окружении на базе Ubuntu в среде WSL2, что обеспечило стабильную сетевую связь с хранилищем данных. В качестве хранилища данных использовалась локально развёрнутая на хост-системе Windows СУБД PostgreSQL 16, которая успешно принимала подключения от инструментов, запущенных в WSL.
Первоначально было создано виртуальное окружение командой python3 -m venv dbt_env, после активации которого выполнена установка пакетов dbt-core==1.8.2 и dbt-postgres==1.8.2. Фиксация версий позволила гарантировать идентичность окружения при последующих запусках. Корректность инсталляции подтверждена выводом команды dbt --version, которая отобразила версии ядра и адаптера.
Следующим шагом стало развёртывание структуры проекта командой dbt init my_data_warehouse. В результате был получен стандартный каркас проекта с каталогами models, seeds, tests, macros, snapshots и файлом конфигурации dbt_project.yml. Для упрощения работы на начальном этапе стандартные примеры моделей из каталога example были удалены, чтобы сосредоточиться на разработке собственных артефактов.
Для обеспечения взаимодействия с базой данных в домашнем каталоге пользователя (~/.dbt/) был создан файл profiles.yml. В нём были прописаны параметры подключения к базе данных lab_database, развёрнутой на локальном сервере PostgreSQL: хост, порт, имя пользователя maria, пароль, имя схемы dbt и дополнительные настройки (keepalives_idle, connect_timeout и threads). Листинг профиля приведен в Приложении 1 настоящего отчета. Предварительно в экземпляре PostgreSQL был создан отдельный пользователь maria и база данных lab_database с полными правами для этого пользователя.
Проверка соединения выполнена командой dbt debug, которая завершилась сообщением Connection test: [OK], что свидетельствовало о корректности сетевых и аутентификационных настроек.
Для наполнения хранилища тестовыми данными в качестве основы для демонстрации были разработаны staging-модели, непосредственно содержащие данные внутри SQL-выражений с помощью конструкции VALUES. Такой подход позволил провести полный цикл трансформации без дополнительной загрузки внешних файлов.
Задача 2. Разработка скриптов автоматизированной работы с данными.
Архитектура проекта была реализована в соответствии с принятой в dbt практикой разделения на слои: staging (очистка и первичная нормализация данных), intermediate (промежуточные объединения и бизнес-логика) и marts (конечные витрины для аналитиков). Соответствующие подкаталоги были созданы внутри models. 
На уровне staging были разработаны две модели: stg_orders.sql и stg_order_items.sql. Первая извлекает данные о заказах с явным приведением даты к типу date, переименованием полей и фильтрацией тестового набора. Вторая содержит информацию о позициях заказов, вычисляя стоимость строки как произведение цены на количество. Обе модели используют временные структуры VALUES, эмулирующие таблицы источников, что делает проект автономным и легко воспроизводимым.
В промежуточном слое была создана модель int_order_items.sql, которая ссылается на stg_order_items. Благодаря конфигурации ephemeral эта модель не материализуется физически, а внедряется в запросы как обобщённое табличное выражение, оптимизируя итоговые запросы.
На уровне marts развёрнуты два объекта: измерение dim_customers.sql, содержащее одно тестовое значение клиента, и таблица фактов fct_sales.sql. Последняя соединяет заказы, клиентов и позиции заказов, вычисляя итоговую сумму продаж и количество уникальных товаров в каждом заказе с помощью агрегатных функций sum и count distinct.
Настройка материализации была зафиксирована непосредственно в файле dbt_project.yml: для staging задано view, для intermediate – ephemeral, а для marts – table. Такой подход позволяет сохранять высокую производительность конечных запросов и при этом не расходовать дисковое пространство на промежуточные представления.
Для гарантии качества данных в каталоге marts был создан файл schema.yml, описывающий тесты для ключевого поля order_id модели fct_sales – проверки на уникальность и отсутствие пропущенных значений. Полный код YAML-файла приведён в Приложении 2.
После завершения разработки выполнена последовательность команд dbt run и dbt test. Первая материализовала все четыре модели, вторая подтвердила отсутствие нарушений целостности данных. Генерация документации осуществлена командой dbt docs generate, которая создала статический веб-сайт с линейной схемой моделей и подробным описанием колонок.
Задача 3. Поставить работу скриптов на расписание.
Оркестрация полученного конвейера была выполнена с помощью Apache Airflow версии 2.9.3, развёрнутого в отдельном виртуальном окружении airflow_env на базе той же операционной системы (WSL Ubuntu). В качестве сервера метаданных Airflow использована отдельная база данных airflow_db в том же экземпляре PostgreSQL, но с выделенным пользователем airflow_user, что способствует изоляции системных таблиц.
После установки apache-airflow==2.9.3 и драйвера psycopg2-binary была выполнена инициализация домашней директории переменной AIRFLOW_HOME. В основном конфигурационном файле airflow.cfg изменены ключевые параметры: строка подключения к базе метаданных через psycopg2 и явное указание SequentialExecutor, наиболее подходящего для учебной среды. Создание учётной записи администратора (admin /admin) обеспечило доступ к веб-интерфейсу.
Далее в стандартном каталоге dags был сформирован файл dbt_dag.py (Приложение 3). В нём описан объект DAG с идентификатором dbt_data_warehouse_pipeline и ежедневным расписанием (@daily), а также три задачи типа BashOperator, которые последовательно выполняют команды dbt run, dbt test и dbt docs generate. Последовательность задач строго определена оператором >>: запуск тестов только после успешной материализации, а генерация документации — после положительного результата тестирования.
Для запуска пайплайна были открыты два терминала с активированным окружением airflow_env и переменной AIRFLOW_HOME. В первом терминале запущен веб-сервер (airflow webserver -p 8081), во втором — планировщик (airflow scheduler). После перехода по адресу http://localhost:8081 и авторизации DAG был активирован переключателем и запущен вручную кнопкой Trigger DAG.
На вкладке Graph View все три задачи (dbt_run, dbt_test, dbt_docs_generate) последовательно окрасились в зелёный цвет, что подтвердило корректное выполнение скриптов. Логи задачи dbt_run показали успешное создание четырёх моделей, dbt_test — отсутствие ошибок в проверках, а dbt_docs_generate — генерацию актуальной документации. Установленное расписание гарантирует автоматическое повторение этого цикла ежедневно без вмешательства человека.

    o.order_id,
    o.order_date,
    c.customer_id,
    c.customer_name,
    sum(oi.line_total) as total_amount,
    count(distinct oi.product_id) as unique_products
from {{ ref('stg_orders') }} o

Приложение 4. Листинг DAG-файла dbt_dag.py для Apache Airflow
from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.bash import BashOperator

DBT_PROJECT_DIR = "/home/maria/dbt_projects/my_data_warehouse"
DBT_EXE = "/home/maria/dbt_env/bin/dbt"

default_args = {
    'owner': 'student',
    'depends_on_past': False,
    'start_date': datetime(2026, 1, 1),
    'retries': 1,
    'retry_delay': timedelta(minutes=5),
}

dag = DAG(
    'dbt_data_warehouse_pipeline',
    default_args=default_args,
    description='Ежедневное обновление витрин dbt',
    schedule_interval='@daily',
    catchup=False,
    tags=['dbt'],
)

dbt_run = BashOperator(
    task_id='dbt_run',
    bash_command=f'cd {DBT_PROJECT_DIR} && {DBT_EXE} run',
    dag=dag,
)

dbt_test = BashOperator(
    task_id='dbt_test',
    bash_command=f'cd {DBT_PROJECT_DIR} && {DBT_EXE} test',
    dag=dag,
)

dbt_docs = BashOperator(
    task_id='dbt_docs_generate',
    bash_command=f'cd {DBT_PROJECT_DIR} && {DBT_EXE} docs generate',
    dag=dag,
)

dbt_run >> dbt_test >> dbt_docs
